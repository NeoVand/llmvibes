## What a modern browser native LLM course should emulate

Recent open technical reports point in the same direction: the modern pipeline is **multi-stage**. DeepSeek-V3 describes large-scale pre-training followed by **supervised fine-tuning and reinforcement learning**. Qwen2.5 explicitly describes improvements in both pre-training and post-training, including **over 1 million SFT samples** and **multistage reinforcement learning**. Kimi K2 describes a **multi-stage post-training process**, including large-scale synthetic data generation and a joint RL stage with real and synthetic environments. DeepSeek-R1 goes even further and argues that strong reasoning behaviors can emerge from **pure reinforcement learning on verifiable tasks**, while the Nature version describes a multistage framework mixing rejection sampling, RL, and SFT. ţłÇciteţłéturn19view0ţłéturn20view0ţłéturn19view1ţłéturn19view2ţłéturn2search6ţłü

Architecturally, the frontier is split between a modern dense decoder recipe and large-scale MoE systems. DeepSeek-V3 and Kimi K2 are both **Mixture-of-Experts** models, and DeepSeek-V3 also emphasizes **multi-token prediction** and load-balancing tricks. But modern dense models remain highly relevant: Meta says Llama 3 keeps a relatively standard decoder-only transformer and adopts **grouped-query attention**, while the Qwen3 report describes a dense stack using **GQA, SwiGLU, RoPE, and RMSNorm with pre-normalization**. For a teaching site, that suggests a clear strategy: teach the dense stack as the core, then add MoE and other frontier tricks as later chapters rather than making routing the entry point. ţłÇciteţłéturn19view0ţłéturn19view1ţłéturn18search0ţłéturn26search4ţłü

For alignment, you also do not need to act as though PPO-style classic RLHF is the only ÔÇťrealÔÇŁ route. DPO was introduced precisely to simplify preference tuning by optimizing on pairwise preferences without separately training a reward model. At the same time, current reasoning work has moved heavily toward **RL with verifiable rewards**. DeepSeekMath introduces **GRPO**, a memory-friendlier RL variant that improved mathematical reasoning, and DeepSeek-R1 explicitly frames verifiable domains such as math and coding as fertile ground for RL-driven reasoning. That is good news for a course project, because verifiable toy tasks are exactly where a browser-native teaching system can shine. ţłÇciteţłéturn2search3ţłéturn3search0ţłéturn19view2ţłü

## Which toy problems are actually good candidates

The strongest general-purpose candidate is **browser-native code generation**, ideally narrowed to a visible and sandboxable subset such as short JavaScript functions or tiny programs that manipulate strings, tables, grids, or canvas scenes. This domain has unusually good public support for the whole pipeline. **HumanEval** evaluates functional correctness from docstrings with unit tests. **MBPP** contains **974** short programming tasks and was used to study models from **244M to 137B** parameters. **APPS** contains **10,000** programming problems and evaluates generated code by running test cases. **CodeContests**, the dataset used for AlphaCode, was released with extensive tests. For preferences, **CodeUltraFeedback** provides **10,000** coding instructions with four responses each and ranked feedback, and recent work such as **CodeDPO** shows how to build preference learning loops from self-generation and validation. That is the cleanest public path from pretraining to SFT to DPO to RLVR. ţłÇciteţłéturn7search0ţłéturn7search1ţłéturn21search1ţłéturn21search16ţłéturn13search2ţłéturn13search17ţłéturn13search0ţłéturn13search15ţłéturn13search1ţłü

The reason code becomes especially attractive for your site is that it can also be made **highly visual**. The browser already gives you a natural execution environment for JavaScript, and if you want Python as a side path, Pyodide brings Python plus packages like NumPy, pandas, SciPy, Matplotlib, and scikit-learn into the browser via WebAssembly. That means the ÔÇťworldÔÇŁ can be a visible one: a generated function lights up test results, transforms a grid, changes a chart, or renders a tiny canvas output. Unlike a pure text benchmark, the user can see the output, the failing test, the corrected trace, and the final verified reward in a single panel. ţłÇciteţłéturn10search2ţłéturn10search6ţłü

The closest runner-up is **text-to-SQL over toy databases**. This is arguably the most ÔÇťLLM-pipeline-faithfulÔÇŁ public task after code. It has an older RL lineage through **Seq2SQL**, which used query execution rewards; public SFT datasets such as **WikiSQL** with **80,654** annotated question-SQL pairs and **Spider** with **10,181** questions over **200** databases; newer real-world data such as **BIRD**; and now a genuinely modern RLVR story through **Arctic-Text2SQL-R1**, which uses a lightweight reward based solely on execution correctness. There is even recent work showing how to make **DPO** work better for text-to-SQL by augmenting it with chain-of-thought style reasoning. If your top priority were fidelity to public literature rather than fun, text-to-SQL would be a superb default. ţłÇciteţłéturn16search0ţłéturn14search4ţłéturn16search1ţłéturn14search8ţłéturn16search2ţłéturn14search0ţłéturn14search18ţłü

## The strongest recommendation

If I were choosing one primary toy problem for this site, I would choose **a tiny code model that writes browser-native programs for visible micro-worlds**.

Concretely, the modelÔÇÖs job would be: given a short natural-language prompt plus a few examples, generate a short JavaScript program or function that transforms a grid, fills a table, manipulates strings, or renders a tiny canvas result. You then verify it with unit tests, snapshot checks, or pixel-diff checks directly in the browser. This gives you the most complete teaching story of any candidate: next-token pretraining over code and task text, SFT on demonstrations, pairwise preference tuning with public code-preference data or local user votes, and RLVR using execution-based rewards. Public code datasets and preference corpora already exist; the ÔÇťvisible micro-worldÔÇŁ layer is the thin custom domain you add to make the course fun and legible. ţłÇciteţłéturn7search0ţłéturn7search1ţłéturn21search1ţłéturn13search0ţłéturn13search1ţłü

This recommendation also gives you the best **interpretability surface**. Every chapter can visualize something different: tokenizer merges over code symbols, attention over prompt/test/spec tokens, ASTs, execution traces, failing versus passing tests, best-of-N sampling, preference comparisons, reward trajectories, and finally RLVR-driven improvement. The user no longer has to trust the model ÔÇťgot smarterÔÇŁ; they can watch a broken program fail, inspect why, and see the post-trained model fix it. That kind of visible causality is harder to achieve in pure chat and less complete in chess or ARC alone. Kimi K2ÔÇÖs report emphasizes synthetic agentic data and joint RL from environments; DeepSeek-R1 emphasizes verifiable RL for reasoning; open code alignment work now uses preference learning and self-validation. ## A model and curriculum that stay modern without breaking the browser

For the core architecture, I would teach a **dense decoder-only transformer** first, using the modern dense recipe rather than an intentionally outdated GPT-2 clone. The cleanest teaching stack is: BPE or unigram tokenizer, RoPE, RMSNorm, SwiGLU, GQA, and a standard autoregressive objective. That reflects where current dense open models are, while keeping the mental model understandable. Then, in later chapters, add **optional frontier modules**: MoE routing, multi-token prediction, synthetic data pipelines, rejection sampling, and GRPO-style RL. Frontier reports justify this split: DeepSeek-V3 and Kimi K2 are MoE-heavy, but Llama 3 and Qwen-family dense models still represent a very modern baseline. ţłÇciteţłéturn19view0ţłéturn19view1ţłéturn18search0ţłéturn26search4ţłü

I would also resist the temptation to make **full 500M-parameter training** the default interactive experience. Adam keeps running averages of the gradient and squared gradient, and a conservative back-of-the-envelope count with fp16 weights and gradients plus Adam moments is about **12 bytes per parameter** before activations. That is roughly **1.12 GiB** of persistent state for **100M** parameters and **5.59 GiB** for **500M** parameters, before you pay for activations, temporary buffers, browser overhead, or the rest of the app. Those numbers do not mean ÔÇťimpossible,ÔÇŁ but they strongly suggest that your live teaching path should focus on smaller dense models, short continued-training bursts, adapters, or precomputed checkpoints rather than pretending users will train a serious 500M model from scratch in one sitting. ţłÇciteţłéturn5search0ţłéturn23calculator1ţłéturn23calculator0ţłü

So the course should probably use **checkpoint waypoints**. Let users truly train very small models end-to-end in browser in the early chapters. Then let them continue-train mid-sized checkpoints or LoRA-style adapters for later chapters. TensorFlow.js explicitly supports building and training models in the browser, while WebGPU provides GPU compute access. Meanwhile, Transformers.js and WebLLM are excellent for loading and running pretrained checkpoints locally, which makes them natural building blocks for the ÔÇťinspect this checkpoint, now fine-tune this adapterÔÇŁ phases of the course. ţłÇciteţłéturn22search1ţłéturn22search3ţłéturn4search3ţłéturn24search1ţłéturn4search2ţłü

A good chapter sequence would be: build the tokenizer and corpus; train a tiny base model; evaluate next-token behavior; add demonstrations for SFT; compare chosen and rejected outputs for DPO; add verifiable environments and GRPO-style RL; then optionally branch into advanced chapters on MoE, multi-token prediction, AI feedback, or self-generated data. That sequence is close enough to the open literature to be honest, but clean enough to remain teachable. ţłÇciteţłéturn19view0ţłéturn20view0ţłéturn2search3ţłéturn3search0ţłéturn3search3ţłü

## How to package it as a reusable vibe engine

The reusable product you want to extractÔÇöthe **vibe engine**ÔÇöshould be the part your current sites already share: a narrative shell, section orchestration, progress state, diagram rendering, persistent local storage, downloadable artifacts, and a pluggable browser execution backend. TerminalVibes proves you can host a local model and a simulated shell inside that shell; GitVibes proves you can swap in a different backend and still preserve the same feel. The new LLM course should treat the actual model playground as another backend, not as a one-off product architecture. ţłÇciteţłéturn1view2ţłéturn1view5ţłü

For this new site, the backend abstraction should be something like **world engine + verifier + checkpoint manager**. In the code-model version, the world engine is the sandboxed interpreter plus renderer; the verifier is unit tests or visual checks; and the checkpoint manager swaps between base, SFT, DPO, and RLVR stages. In the SQL version, the world engine is SQLite-in-Wasm; in the chess version, it is the board plus Stockfish-in-Wasm. The surrounding course shell can stay almost identical.
