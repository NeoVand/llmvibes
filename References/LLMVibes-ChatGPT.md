## What a modern browser native LLM course should emulate

Recent open technical reports point in the same direction: the modern pipeline is **multi-stage**. DeepSeek-V3 describes large-scale pre-training followed by **supervised fine-tuning and reinforcement learning**. Qwen2.5 explicitly describes improvements in both pre-training and post-training, including **over 1 million SFT samples** and **multistage reinforcement learning**. Kimi K2 describes a **multi-stage post-training process**, including large-scale synthetic data generation and a joint RL stage with real and synthetic environments. DeepSeek-R1 goes even further and argues that strong reasoning behaviors can emerge from **pure reinforcement learning on verifiable tasks**, while the Nature version describes a multistage framework mixing rejection sampling, RL, and SFT. Architecturally, the frontier is split between a modern dense decoder recipe and large-scale MoE systems. DeepSeek-V3 and Kimi K2 are both **Mixture-of-Experts** models, and DeepSeek-V3 also emphasizes **multi-token prediction** and load-balancing tricks. But modern dense models remain highly relevant: Meta says Llama 3 keeps a relatively standard decoder-only transformer and adopts **grouped-query attention**, while the Qwen3 report describes a dense stack using **GQA, SwiGLU, RoPE, and RMSNorm with pre-normalization**. For a teaching site, that suggests a clear strategy: teach the dense stack as the core, then add MoE and other frontier tricks as later chapters rather than making routing the entry point. ## Which toy problems are actually good candidates

The strongest general-purpose candidate is **browser-native code generation**, ideally narrowed to a visible and sandboxable subset such as short JavaScript functions or tiny programs that manipulate strings, tables, grids, or canvas scenes. This domain has unusually good public support for the whole pipeline. **HumanEval** evaluates functional correctness from docstrings with unit tests. **MBPP** contains **974** short programming tasks and was used to study models from **244M to 137B** parameters. **APPS** contains **10,000** programming problems and evaluates generated code by running test cases. **CodeContests**, the dataset used for AlphaCode, was released with extensive tests. For preferences, **CodeUltraFeedback** provides **10,000** coding instructions with four responses each and ranked feedback, and recent work such as **CodeDPO** shows how to build preference learning loops from self-generation and validation. That is the cleanest public path from pretraining to SFT to DPO to RLVR. The reason code becomes especially attractive for your site is that it can also be made **highly visual**. The browser already gives you a natural execution environment for JavaScript, and if you want Python as a side path, Pyodide brings Python plus packages like NumPy, pandas, SciPy, Matplotlib, and scikit-learn into the browser via WebAssembly. That means the ŌĆ£worldŌĆØ can be a visible one: a generated function lights up test results, transforms a grid, changes a chart, or renders a tiny canvas output. ## The strongest recommendation

If I were choosing one primary toy problem for this site, I would choose **a tiny code model that writes browser-native programs for visible micro-worlds**.

Concretely, the modelŌĆÖs job would be: given a short natural-language prompt plus a few examples, generate a short JavaScript program or function that transforms a grid, fills a table, manipulates strings, or renders a tiny canvas result. You then verify it with unit tests, snapshot checks, or pixel-diff checks directly in the browser. This gives you the most complete teaching story of any candidate: next-token pretraining over code and task text, SFT on demonstrations, pairwise preference tuning with public code-preference data or local user votes, and RLVR using execution-based rewards. Public code datasets and preference corpora already exist; the ŌĆ£visible micro-worldŌĆØ layer is the thin custom domain you add to make the course fun and legible. ## A model and curriculum that stay modern without breaking the browser

For the core architecture, I would teach a **dense decoder-only transformer** first, using the modern dense recipe rather than an intentionally outdated GPT-2 clone. The cleanest teaching stack is: BPE or unigram tokenizer, RoPE, RMSNorm, SwiGLU, GQA, and a standard autoregressive objective. That reflects where current dense open models are, while keeping the mental model understandable. Then, in later chapters, add **optional frontier modules**: MoE routing, multi-token prediction, synthetic data pipelines, rejection sampling, and GRPO-style RL. Frontier reports justify this split: DeepSeek-V3 and Kimi K2 are MoE-heavy, but Llama 3 and Qwen-family dense models still represent a very modern baseline. ŅłĆciteŅłéturn19view0Ņłéturn19view1Ņłéturn18search0Ņłéturn26search4Ņłü

I would also resist the temptation to make **full 500M-parameter training** the default interactive experience. Adam keeps running averages of the gradient and squared gradient, and a conservative back-of-the-envelope count with fp16 weights and gradients plus Adam moments is about **12 bytes per parameter** before activations. That is roughly **1.12 GiB** of persistent state for **100M** parameters and **5.59 GiB** for **500M** parameters, before you pay for activations, temporary buffers, browser overhead, or the rest of the app. Those numbers do not mean ŌĆ£impossible,ŌĆØ but they strongly suggest that your live teaching path should focus on smaller dense models, short continued-training bursts, adapters, or precomputed checkpoints rather than pretending users will train a serious 500M model from scratch in one sitting. ŅłĆciteŅłéturn5search0Ņłéturn23calculator1Ņłéturn23calculator0Ņłü

So the course should probably use **checkpoint waypoints**. Let users truly train very small models end-to-end in browser in the early chapters. Then let them continue-train mid-sized checkpoints or LoRA-style adapters for later chapters. TensorFlow.js explicitly supports building and training models in the browser, while WebGPU provides GPU compute access. Meanwhile, Transformers.js and WebLLM are excellent for loading and running pretrained checkpoints locally, which makes them natural building blocks for the ŌĆ£inspect this checkpoint, now fine-tune this adapterŌĆØ phases of the course. A good chapter sequence would be: build the tokenizer and corpus; train a tiny base model; evaluate next-token behavior; add demonstrations for SFT; compare chosen and rejected outputs for DPO; add verifiable environments and GRPO-style RL; then optionally branch into advanced chapters on MoE, multi-token prediction, AI feedback, or self-generated data. That sequence is close enough to the open literature to be honest, but clean enough to remain teachable. ŅłĆciteŅłéturn19view0Ņłéturn20view0Ņłéturn2search3Ņłéturn3search0Ņłéturn3search3Ņłü

## How to package it as a reusable vibe engine

The reusable product you want to extractŌĆöthe **vibe engine**ŌĆöshould be the part your current sites already share: a narrative shell, section orchestration, progress state, diagram rendering, persistent local storage, downloadable artifacts, and a pluggable browser execution backend. TerminalVibes proves you can host a local model and a simulated shell inside that shell; GitVibes proves you can swap in a different backend and still preserve the same feel. For this new site, the backend abstraction should be something like **world engine + verifier + checkpoint manager**. In the code-model version, the world engine is the sandboxed interpreter plus renderer; the verifier is unit tests or visual checks; and the checkpoint manager swaps between base, SFT, DPO, and RLVR stages. In the SQL version, the world engine is SQLite-in-Wasm; in the chess version, it is the board plus Stockfish-in-Wasm. The surrounding course shell can stay almost identical.
